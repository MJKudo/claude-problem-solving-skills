# 工程別成果物テンプレート

各工程の成果物ファイルは、該当テンプレートをコピーして作成する。`合意ステータス` は draft → reviewed(独立レビュー完了)→ agreed(クライアント合意)と遷移させ、agreed になるまで次工程に進んではならない。「観点(障害)」「観点(改善)」の注記は選択中のモードに該当する側を使い、もう一方は削除する。

事実の出所表記((確認済み)=証拠参照必須/(資料記載)/(推定)/(帰結))を全テンプレートで用いること。

## 課題一覧
観点(障害): 検証済みの原因(証拠参照付き)と未検証仮説を明確に区別する。原因確定に至らない場合は「確定に必要な追加調査」を課題として列挙する
観点(改善): 目標達成の阻害要因(各要因に根拠を付す)

因果を1段ずつ遡って記述する場合(5 Whys 等)は、**各遷移に証拠を添付する**。証拠のない遷移は(推定)とし、原因確定の根拠にしない。**本ワークフローでは回数を停止条件に用いず**、証拠がこれ以上得られなくなった時点を停止点とする(運用規範)。

主要な論点では競合する仮説を2つ以上並置し、各仮説に反証条件(これを否定する観測は何か)を付して、反証を試みてから採否を判断する。原因が単一の要因、特に単一の「人的ミス」に収束したら掘り下げ不足のサインとみなし、複数の寄与要因(単独では不十分だが結合して障害を起こす要因)を疑う。

# 対処要件定義

合意ステータス: draft

## 前提(工程3で合意した課題の要約)

## 対処の要件
観点(障害): 対処策の要件 — 修正方針・影響範囲・検証方法
観点(改善): 改修・構築の要件 — 構成・フレームワーク・データスキーマ等

## 対策強度の評価(障害モードでは必須)
(原因究明の対策は、放置すると「教育・注意喚起・手順の再徹底」といった**弱い対策に偏る**ことが、医療安全分野の調査で報告されている〔技術システムでの再現は未確認〕。この偏りを避けるため、本ワークフローの運用規範として、各対策を下表の強度で分類し、**弱い対策のみで構成された案は不採用とする**。弱い対策を含めてよいが、少なくとも1つは中〜強の対策を伴わせる)

| 強度 | 内容 | 例 |
| ---- | ---- | -- |
| 弱 | 人の注意・記憶・遵守に依存する | 注意喚起、教育・研修、手順書の再徹底、チェックリストの追加 |
| 中 | 誤りを検知・警告する。誤りは起こりうるが被害を抑える | 監視・アラート、入力検証、二重確認の仕組み化、自動テストの追加 |
| 強 | 設計・仕組みによって、そもそも起こらなくする | 設計変更、インターフェースの物理的・論理的な排他、依存の除去、自動化による人手介在の廃止 |

| # | 対策 | 対応する課題(工程3の#) | 強度(弱/中/強) | 選定理由 |
| - | ---- | ------------------------ | ---------------- | -------- |
