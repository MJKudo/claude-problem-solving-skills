# 調査・検証ガイド

共通サイクルのステップ2「調査・検証」はこのガイドに従う。

## 手段の使い分け

1. **資料の把握と突合**(工程1では最初に実施): クライアントの提示資料(設計書・仕様書・構成図・手順書等)を収集・把握する。「提示資料はすべて正しい」という前提はほとんどの場合破綻しているものとして扱い、資料由来の情報は(資料記載)と表記する。実態(コード・設定・実機・ログ)との突合で裏取りされた時点で(確認済み)に昇格させる。乖離が見つかった場合は乖離自体を成果物に記録する — 乖離は原因究明の有力な手がかりになる
2. **直接調査**: 対象(コード・ログ・設定・DB 等)に作業環境からアクセスできる場合は Claude が自ら調査する。必要に応じ WebSearch で既知事象・製品仕様・類似事例を調査する
3. **切り分けタスク提示**: 直接アクセスできない場合(装置・本番環境・別ネットワーク等)は、下記書式の切り分けタスクリスト+結果入力フォーマットを提示し、ユーザーの実施結果を証拠として受け取る

判断基準: 「Claude が今この場で読める・実行できるか」。読めるなら直接調査、読めないなら切り分けタスク提示。混在してよい(コードは直接・実機はタスク)。

## 仮説検証の記録様式

仮説は必ず次の形式で成果物の「調査・検証の記録」に残す。検証されるまで(推定)扱いとする。

| 仮説 | この仮説が正しければ観測されるはずのこと | 検証手段 | 結果 | 判定 |
| ---- | ------------------------------------ | -------- | ---- | ---- |

判定は 支持 / 棄却 / 未検証 のいずれか。棄却された仮説も削除せず残す(再訪の防止)。

## 原因分析手法の使い分け(障害モードの原因特定)

### 0. 前提として知っておくべきこと(手法の限界)

これらの手法は「使えば正解に至る道具」ではない。学術文献の検証では、**これらの手法が有効であることを示す実証は乏しく、逆に限界を示す実証は複数存在する**。手法は下記のガードレール付きでのみ用いる。

- **根本原因分析(RCA)全般**: 有効性(再発防止に効くこと)を実証した研究は確認されていない。また RCA の提案対策は「教育・注意喚起・手順の再徹底」といった**弱い対策に偏る傾向**が実証されている。「根本原因」という言葉自体が「原因は一つ(または少数)である」という誤った前提を助長する、という批判がある
- **5 Whys**: **調査者の既有知識の外にある原因には到達できない**。また**同じ問題を別のチームが分析すると別の原因に到達する**(再現性が低い)ことが、推進側のガイド自身に明記されている
- **FMEA**: 同じ対象を2つの独立チームが分析したとき、**列挙された故障モードの一致率は 17%** だったという実証がある。また FMEA チームが付けたスコアは、実際のインシデント記録と有意に相関しなかった
- **FTA**: **ソフトウェアの挙動を古典的な「故障」として扱えない**ため、ソフトウェア起因の不具合には適用できない(Leveson 2004)

(根拠と出典: `docs/superpowers/specs/2026-07-13-rca-fta-fmea-verification-report.md`。なお上記の実証の多くは医療安全分野のものであり、装置・Web システムへの外的妥当性は検証されていない。)

### 1. 仮説生成の前に: 既知問題との照合(必須)

原因仮説を自分で組み立てる前に、**既に知られている問題の知識を先に使う**。これを飛ばして推論から始めると、既知の不具合を一から「発見」し直すことになる。

- 過去の同種案件・障害記録・既知の不具合パターン
- 製品・ライブラリ・OS の既知不具合(リリースノート、Issue、アドバイザリ。WebSearch で確認する)
- 直近の変更履歴(デプロイ・設定変更・部品交換・アップデート)との時間的な一致

照合結果は、一致がなかった場合も含めて工程3の成果物に記録する。

### 2. 手法の選択

単純な案件では 5 Whys(下記のガードレール付き)のみで十分なことが多い。複数の寄与要因が疑われる複雑な案件でのみ、目的に応じて以下を組み合わせる。

- **原因候補を広く洗い出す(発散)**: 特性要因図(fishbone。問題を魚の頭に見立て、人・機械・方法・材料・環境等のカテゴリ別に原因候補を骨のように書き出す図)
- **トップダウンに故障経路を辿る(演繹)**: フォールトツリー(FTA。上位の障害から出発し、それを引き起こしうる下位の事象へ、「両方必要(AND)」か「どちらかで足りる(OR)」かを区別しながら枝分かれさせて遡る手法)
  - **適用制限(必須)**: 不具合が**ソフトウェア起因**である場合、FTA を主たる分析手法として用いてはならない。ソフトウェアの挙動は「部品が壊れる」型の故障として表現できないため、検証可能な基本事象が置けず、木が成立しない
  - **禁止事項**: 「ソフトウェアが誤動作する」のような、**検証も対策もできない粒度の基本事象(葉ノード)を置いてはならない**。各基本事象は、観測・検証できる具体的な故障状態でなければならない
- **個々の部品・機能の故障が上位へ与える影響を網羅する**: FMEA(部品・機能ごとに「どう壊れうるか」を洗い出し、それぞれが上位に与える影響を一覧化する手法)
  - **スコア単独判定の禁止(必須)**: 重大度・発生確率・検出可能性等のスコア(RPN 等)を、課題の優先順位や採否の**唯一の根拠にしてはならない**。必ず実際の障害ログ・インシデント記録との突合を伴わせる
  - **単一チーム出力を信用しない**: 列挙結果は分析主体に強く依存する。独立レビューでは体裁確認ではなく、**別視点からの故障モードの再列挙と差分照合**を行い、差分が大きい場合は draft に差し戻す
- **1本の因果連鎖を深掘りする**: 5 Whys(「なぜ」を繰り返して原因を1段ずつ掘る手法)
  - **証拠添付の必須化**: 各「なぜ」の遷移(1段掘るごと)に**証拠を添付する**。証拠のない遷移は(推定)とし、原因確定の根拠にしない
  - **複数系列の並行展開**: 単一の因果連鎖に早期収束させない。主要な論点では独立した「なぜ」の系列を2本以上並行して展開する
  - **停止条件**: **本ワークフローでは回数を停止条件に用いない**。**証拠がこれ以上得られなくなった時点**を停止点とする(これは文献的裏付けのある知見ではなく、**本ワークフローの運用規範**である。「5回という回数が目安にすぎない」という主張自体は、検証の結果「引用の裏付け範囲を超える」として不採用になっている)
  - **知識の天井の自覚**: 自分(および分析チーム)の既有知識の外に原因がある可能性を常に残す。行き詰まったら、より体系的な手法(FTA・FMEA・特性要因図)や外部情報(既知不具合・製品仕様)へエスカレーションする
- いずれの手法で導いた原因も、仮説検証の記録様式で裏取りする。定性的な判断だけで確定させない

## 「なぜ」より「どのように」(人・運用が絡む障害)

人の操作や運用手順が関与する障害では、「なぜ間違えたか」と原因を人に求めると、局所合理性(その場でなぜその判断が妥当に見えたか)を見落とす。次の問いに置き換える:

- その時点で何を観測していたか(どんな情報・表示・ログが見えていたか)
- 何を期待・目標としていたか
- どういう知識・手順に基づいて判断したか
- どうコミュニケーションしていたか

関与者はその時点の最善の情報で善意に基づき行動したと仮定して検証する(ブレームレス)。「人的ミス」で分析を止めず、それをより深いシステム上の要因の症状として扱う。

## 切り分けタスクリストの書式

各タスクは次を必ず含む:

1. **復元手段の確保**(先頭に必須): 実施前に取るバックアップ・スナップショット・設定の控え・現状の記録。取れない場合は「実施内容・条件・結果を記録する手順」を代わりに置く。どちらも不可能なタスクは提示しない
2. **実施手順**: 番号付きの具体的手順(コマンド・操作・観察点)
3. **結果入力フォーマット**: ユーザーが埋めるだけでよい形(例: 「手順3のエラーメッセージ: ____」「LED の色: 赤 / 緑 / 消灯」)
4. **所要目安と中断条件**: 想定外の状態になったら中断して報告する旨

## 本番環境の保護(絶対規則)

- 稼働中の本番環境に対して、Claude が直接、不可逆な操作(データの削除・上書き、設定変更、再起動・停止、デプロイ等)を行うことは**いかなる場合も禁止**
- **環境が本番かどうか不明な場合は本番とみなす**
- 本番環境への操作が必要な場合は、復元手段の確保を先頭に置いた手順書として提示し、実施はユーザーが行う
- 不可逆操作の判定: 「元に戻す手順を、実行前のこの場で具体的に書けるか」。書けなければ不可逆として扱う
- 本番以外でも、状態を変えうる操作は実施前にその旨と復元手段を明示してユーザーの許可を得る

## evidence/ への保存規約

- 分量のある証拠(ログ抜粋・計測結果・切り分けタスクの実施結果・資料との乖離の対照表等)は `docs/system-analysis/<案件slug>/evidence/` に保存する
- ファイル名: `<連番2桁>-<内容の短い英語ケバブケース>.md`(例: `01-error-log-extract.md`)
- 工程成果物からは必ずファイル名で参照する(「(確認済み: evidence/01-error-log-extract.md)」)
- 証拠ファイル冒頭に、いつ・どこから・どうやって取得したかを1〜3行で記す
