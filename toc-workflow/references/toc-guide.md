# TOC ツール構築ガイド

工程でツリー構築・制約特定を行うとき、本ガイドの該当節を読んでから着手する。用語の定義は SKILL.md の用語表に従う。

## 1. UDE の書き方(事実性チェックリスト)

UDE として認める前に全項目を確認する:

- [ ] 観測可能か(誰かが見た・測った事実か。意見・解釈ではないか)
- [ ] 1文で単一の事象を述べているか(複合文は分割する)
- [ ] 解決策の裏返しでないか(「〜がない」「〜すべきなのにしていない」は不可。「無いこと」ではなく「起きている困った事実」に言い換える)
- [ ] 頻度・程度・影響を問われたら答えられるか(可能なら定量化する)

言い換え例: 「進捗共有の仕組みがない」(不可)→「納期遅延が四半期に3件発生し、いずれも発覚が納期1週間前だった」(可)

## 2. ルートの選択(古典法 / 3クラウド法)

| ルート | 手順 | 向いている状況 |
| ------ | ------ | -------------- |
| 古典法 | UDE 一覧→現状ツリーで因果を遡る→少数の根本原因→中核対立を言語化 | UDE 間の因果が見えやすい。データ・記録が揃っている |
| 3クラウド法 | 代表的な UDE を3件選ぶ→各 UDE を個別クラウドにする→3つを見比べて共通の型を統合クラウドにまとめる→中核対立を得る→検証用の現状ツリーを統合クラウドから構築する | 関係者間の対立色が強い。因果の記録が乏しく、立場の異なる関係者の合意形成が重要 |

どちらのルートでも工程2の完了条件は同じ(中核対立または少数の根本原因と、現状構造への合意)。

## 3. クラウドの作り方

1. 対立する2つの行動を D と D' に置く(どちらも現に誰かが取っている・取ろうとしている行動)
2. それぞれ「その行動は何の要件を満たすためか」を B・C に書く
3. B と C の両方が必要とする共通目標を A に書く
4. 各矢印(A←B、A←C、B←D、C←D'、D↔D')の背後の「なぜならば」(前提)をすべて列挙する
5. 前提ごとに「本当に常に成り立つか。成り立たない条件を作れないか」を問い、崩せる前提にインジェクションを導出する

注意: D・D' はどちらも正当な要件から出た行動として扱う。片側を誤りと断じない。目指すのは妥協(両方を少しずつ諦める)ではなく、前提の無効化による対立の消滅である。

## 4. 未来ツリーとネガティブブランチ

1. インジェクションと現実の条件を起点に「〜ならば〜になる」を積み上げる
2. 工程1の各 UDE がどの望ましい事象に置き換わるかの対応を確認する(置き換わらない UDE が残る場合、インジェクションが不足している)
3. ネガティブブランチ: 「このインジェクションで新たに困ることは何か」を必ず問い、悪影響への因果経路を書く。経路が現実的なら、それを断つ対策(追加のインジェクション)を足して刈り込む
4. インジェクション〜主要な望ましい事象のコア経路は CLR 8観点で逐矢印検証し、記録を残す

## 5. 前提条件ツリーと移行ツリー

1. 「この解決策を実行できない理由・障害は何か」を関係者から集める(障害の列挙は抵抗ではなく貢献として扱う)
2. 各障害に「その障害を乗り越えた状態」=中間目標を対応させる
3. 中間目標を依存順に並べる(どれが先に達成されていないと次に進めないか)
4. 手順への具体化で迷う中間目標がある場合のみ、移行ツリー(アクション→期待される状態→次のアクション)まで作る。単純な案件では実行計画表への直接展開でよい

## 6. 5集中ステップの進め方

- **制約の特定**: 滞留(仕掛かりが積み上がる場所)・リードタイム・稼働率・残業の集中などのデータを取る。データが取れない場合は現場観察で代替し、観察事実を記録する
- **徹底活用の定石**: 制約の時間を奪っているもの(手待ち・段取り・会議・不良品の処理)を除去する。制約に不良品を流さない(前工程で検査する)。制約の休止時間を減らす
- **従属の定石**: 非制約の効率を追わない(制約より速く作っても在庫が増えるだけ)。制約の直前に時間的余裕(バッファ)を置き、制約を止めない。投入を制約のペースに合わせて制御する
- **強化の判断**: 活用・従属で足りるかを先に確認する。足りない場合のみ、能力増強(設備・人員・外注)を投資対効果で設計する
- **惰性防止**: 制約が移動・解消したら、旧制約に合わせて作った方針・ルール・バッファを必ず見直す(見直さない方針が次の制約になる)

## 7. CLR 8観点の定義と適用例

| 観点 | 問い | 適用例 |
| ------ | ------ | -------- |
| 明瞭性 | 記述は誰が読んでも同じ意味に取れるか | 「品質が悪い」→ 何の・どの指標か明確にする |
| 実体の存在 | その事象は本当に存在するか(観測できるか) | UDE の事実性チェックと同じ |
| 因果の存在 | 「AならばB」のつながりは本当にあるか | A を消したら B も消えるかを問う |
| 原因の不十分性 | A だけで B が起きるか。他に必要な条件はないか | 「AかつCならばB」に書き直す |
| 別の原因 | B を起こす別の原因は存在しないか | 競合仮説を並置し反証条件を付す |
| 因果の逆転 | 原因と結果が逆ではないか | どちらが先に観測されるかを確認する |
| 予測効果の存在 | A が真なら B 以外にも観測されるはずの事象があるか。それは観測されているか | 予測される副次効果を探して裏取りする |
| 循環論法 | B の存在を A の唯一の根拠にしていないか | A の独立した根拠を求める |

## 8. 方針制約を見つけたとき

制約が物理的な能力ではなく、方針・ルール・評価指標(例: 部分最適の効率評価、ロットサイズの慣行、承認の階層)である場合:

- 能力増強(強化)では解消しない。方針を変える必要がある
- 方針の背後には「その方針を必要とした過去の対立」があることが多い。思考プロセスモード(TP)への切替、または該当方針をクラウドにかけて前提を崩すことをクライアントに提案する
- 切替時は工程2までの成果物(制約認定)を思考プロセスモード(TP)の UDE・現状構造の入力として引き継ぐ
- 切替時は `01-problem-structure.md` 冒頭のモード欄を更新する(中断→再開時に旧モードで復元されるのを防ぐ)
